# UI 상세 테스트 케이스

## 목적

테스터가 동일한 절차로 수동 검증을 수행하고, 실패를 재현 가능한 수준으로 기록할 수 있도록 핵심 UI 시나리오를 절차 중심으로 정리합니다.

## 공통 환경

- 애플리케이션 실행: `npm run dev`
- 기본 브라우저: Chromium 계열 권장
- 기본 저장 경로 예시: `/tmp/depssmuggler-manual`
- SMTP 테스트용 예시 값:
  - Host: `smtp.example.com`
  - Port: `2525`
  - From/User: `sender@example.com`
  - Password: `app-password`
  - To: `offline@example.com`

## UI-WIZ-001 홈에서 패키지 타입 선택 후 위저드 진입

- 목적: 홈 화면 진입 카드가 올바른 위저드 흐름으로 연결되는지 확인
- 사전 조건: 앱이 `/#/`에서 시작된 상태
- 절차:
  1. 홈 화면에서 라이브러리 또는 OS 패키지 진입 카드를 선택합니다.
  2. 위저드 화면으로 이동했는지 확인합니다.
  3. 다른 패키지 타입으로 다시 진입해 이전 선택 상태가 남지 않는지 확인합니다.
- 기대 결과:
  - 선택한 타입에 맞는 제목, 단계, 입력 UI가 표시됩니다.
  - 이전 타입의 배포판/버전/패키지명 상태가 섞이지 않습니다.
- 실패 기준:
  - 잘못된 타입의 입력 필드가 보입니다.
  - 새로 진입했는데 이전 선택값이 그대로 남습니다.

## UI-WIZ-002 일반 패키지 검색 후 장바구니 추가

- 목적: `pip` 또는 `npm` 검색 결과가 장바구니까지 정상 연결되는지 확인
- 사전 조건: `/#/wizard`에서 일반 패키지 타입이 선택된 상태
- 절차:
  1. `pip` 또는 `npm` 패키지 타입을 선택합니다.
  2. 패키지명을 입력하고 검색 결과를 확인합니다.
  3. 버전을 선택한 뒤 장바구니에 추가합니다.
  4. 장바구니 화면으로 이동합니다.
- 기대 결과:
  - 검색 결과 목록이 표시됩니다.
  - 선택한 패키지명, 버전, 타입이 장바구니에 정확히 추가됩니다.
- 실패 기준:
  - 검색 결과가 없는데 로딩만 지속됩니다.
  - 장바구니 항목의 타입 또는 버전이 다르게 기록됩니다.

## UI-DL-001 장바구니에서 일반 다운로드 완료까지 진행

- 목적: 일반 다운로드 플로우의 기본 경로를 smoke 수준으로 검증
- 사전 조건: 장바구니에 일반 패키지 1개 이상 존재
- 절차:
  1. 장바구니에서 `다운로드 시작`을 누릅니다.
  2. 다운로드 화면에서 기본 폴더 경로, 출력 형식, 패키지 수를 확인합니다.
  3. `다운로드 시작`을 눌러 완료 화면까지 진행합니다.
  4. 완료 화면의 전달 방식, 실제 산출물 경로, 후속 액션 버튼을 확인합니다.
- 기대 결과:
  - 설정의 기본 다운로드 폴더와 출력 형식이 반영됩니다.
  - 진행 중 상태가 깨지지 않고 완료 화면이 표시됩니다.
  - 전달 방식이 `로컬 저장`이면 완료 화면에도 동일하게 표시됩니다.
- 실패 기준:
  - 기본 경로 또는 출력 형식이 설정값과 다릅니다.
  - 완료 후 산출물 정보가 비어 있거나 잘못 표시됩니다.

## UI-DL-002 전달 방식 전환

- 목적: 다운로드 완료 후 처리 옵션이 UI와 내부 옵션에 일관되게 반영되는지 확인
- 사전 조건: 다운로드 화면 진입 상태, SMTP 관련 설정 저장 완료
- 절차:
  1. 전달 방식을 `로컬 저장`으로 두고 안내 문구와 입력 상태를 확인합니다.
  2. 전달 방식을 `이메일 전달`로 바꿉니다.
  3. 현재 수신자, 분할 옵션, 첨부 크기 관련 안내가 나타나는지 확인합니다.
  4. 다시 `로컬 저장`으로 전환해 이메일 관련 안내가 사라지는지 확인합니다.
- 기대 결과:
  - 선택한 전달 방식에 맞는 UI만 표시됩니다.
  - 이메일 전달에서는 수신자 및 분할 관련 정보가 즉시 보입니다.
- 실패 기준:
  - 전달 방식과 무관한 입력 필드가 남아 있습니다.
  - 설정의 SMTP 수신자나 파일 분할 기준이 화면에 반영되지 않습니다.

## UI-SET-001 SMTP 설정 저장과 새로고침 복원

- 목적: 설정 저장과 persistence 경로를 검증
- 사전 조건: `/#/settings` 진입 상태
- 절차:
  1. SMTP Host, Port, From, User, Password, To를 입력합니다.
  2. `저장` 버튼을 누르고 성공 메시지를 확인합니다.
  3. 페이지를 새로고침합니다.
  4. 같은 값이 그대로 남아 있는지 확인합니다.
- 기대 결과:
  - 저장 성공 피드백이 즉시 표시됩니다.
  - 새로고침 뒤에도 입력값이 유지됩니다.
- 실패 기준:
  - 일부 필드만 복원되거나 빈 값으로 초기화됩니다.
  - 포트 숫자 값이 문자열/기본값으로 잘못 돌아갑니다.

## UI-SET-002 SMTP 연결 테스트

- 목적: 설정 화면의 SMTP 테스트 진입점을 검증
- 사전 조건: `UI-SET-001`과 동일
- 절차:
  1. SMTP 필수값을 비운 상태에서 `연결 테스트`를 눌러 경고를 확인합니다.
  2. 예시 값을 모두 채우고 다시 `연결 테스트`를 누릅니다.
  3. 결과 메시지와 중복 클릭 방지 상태를 확인합니다.
- 기대 결과:
  - 필수값 누락 시 즉시 오류 또는 안내가 표시됩니다.
  - 정상 입력 시 성공 또는 실패 메시지가 분명히 구분됩니다.
- 실패 기준:
  - 비어 있는 값인데 테스트가 진행됩니다.
  - 결과 메시지가 없거나 이전 결과가 남아 혼동을 일으킵니다.

## UI-HIS-001 이메일 히스토리 재다운로드

- 목적: 히스토리 기반 재다운로드가 저장된 수신자를 복원하는지 확인
- 사전 조건: 이메일 전달 성공 이력이 하나 이상 존재
- 절차:
  1. `/#/history`로 이동합니다.
  2. 이메일 전달 이력의 재다운로드 액션을 실행합니다.
  3. 확인 다이얼로그에서 재다운로드를 승인합니다.
  4. 다운로드 화면에서 현재 수신자와 패키지 정보를 확인합니다.
  5. 다운로드를 다시 시작합니다.
- 기대 결과:
  - 다운로드 화면의 현재 수신자는 히스토리 수신자로 복원됩니다.
  - 전역 설정의 기본 SMTP 수신자 자체는 변경되지 않습니다.
- 실패 기준:
  - 히스토리 수신자가 복원되지 않습니다.
  - 재다운로드 후 설정 화면의 기본 수신자까지 덮어써집니다.

## UI-OS-001 OS 패키지 전용 다운로드 흐름

- 목적: OS 패키지 전용 위저드와 다운로드 화면 분기를 검증
- 사전 조건: `/#/wizard`에서 OS 패키지 타입 진입 가능
- 절차:
  1. OS 패키지 타입을 선택하고 배포판과 패키지명을 입력합니다.
  2. 검색 결과에서 패키지를 선택해 장바구니에 추가합니다.
  3. `OS 다운로드 진행`으로 전용 다운로드 화면에 진입합니다.
  4. 출력 옵션에서 `archive`, `repository`, `both` 중 하나를 선택합니다.
  5. 다운로드를 시작하고 완료 화면의 생성 산출물을 확인합니다.
- 기대 결과:
  - 일반 다운로드 화면이 아니라 OS 전용 다운로드 화면이 표시됩니다.
  - 출력 옵션 변경이 결과 화면 요약에도 반영됩니다.
- 실패 기준:
  - OS 패키지인데 일반 다운로드 플로우로 이동합니다.
  - 생성 산출물 유형이 선택한 출력 옵션과 다릅니다.

## UI-DL-003 빈 장바구니 상태에서 다운로드 페이지 진입

- 목적: 빈 상태 예외 처리를 검증
- 사전 조건: 장바구니가 비어 있음
- 절차:
  1. `/#/download`로 직접 이동하거나 빈 장바구니 상태에서 진입합니다.
  2. 빈 상태 메시지와 이동 버튼을 확인합니다.
  3. 안내 버튼으로 장바구니 또는 위저드로 복귀합니다.
- 기대 결과:
  - 사용자에게 다음 액션이 있는 빈 상태 화면이 표시됩니다.
  - 복귀 액션이 올바른 화면으로 이동합니다.
- 실패 기준:
  - 빈 화면 또는 깨진 레이아웃이 나타납니다.
  - 복귀 버튼이 동작하지 않습니다.

## UI-DL-004 다운로드 취소 및 재시도

- 목적: 장시간 다운로드 또는 실패 상황에서 상태 전이가 일관적인지 확인
- 사전 조건: 실패 또는 취소를 재현할 수 있는 다운로드 항목 준비
- 절차:
  1. 다운로드를 시작한 뒤 가능한 시점에 취소를 실행합니다.
  2. 취소 후 상태, 버튼, 로그 영역을 확인합니다.
  3. 실패 항목이 있는 경우 재시도를 실행합니다.
- 기대 결과:
  - 취소 후 진행률이 멈추고 상태가 취소됨으로 갱신됩니다.
  - 재시도는 실패 항목 또는 전체 재실행 정책에 맞게 동작합니다.
- 실패 기준:
  - 취소 후에도 진행률이 계속 증가합니다.
  - 재시도 버튼이 잘못된 항목에 활성화됩니다.

## UI-CFG-001 캐시 통계 조회와 캐시 정리

- 목적: 설정 화면 또는 캐시 UI의 관리 동작을 검증
- 사전 조건: 캐시 기능 활성화, 캐시가 일부 쌓여 있는 상태 권장
- 절차:
  1. 캐시 통계 조회를 실행합니다.
  2. 총 크기, 엔트리 수, 패키지 타입별 상세 값이 표시되는지 확인합니다.
  3. 캐시 비우기를 실행하고 완료 메시지를 확인합니다.
  4. 통계를 다시 조회합니다.
- 기대 결과:
  - 정리 전후 수치 변화가 사용자에게 보입니다.
  - 정리 후 통계가 0 또는 감소한 값으로 갱신됩니다.
- 실패 기준:
  - 정리 후에도 이전 수치가 그대로 남습니다.
  - 상세 타입 통계와 총합이 일관되지 않습니다.

## 기록 규칙

- 실패 시 다음을 함께 기록합니다.
  - 테스트 케이스 ID
  - 실행 환경과 앱 커밋
  - 실제 결과
  - 기대 결과
  - 재현 절차
  - 스크린샷 또는 콘솔/로그 증거

자동화 전환 우선순위와 mock 전략은 [UI Playwright 전환 시나리오](./ui-testing-playwright-conversion.md)에서 관리합니다.
